1) Contexte & objectifs

Le Campus souhaite un outil de gestion de tickets pour déclarer et suivre des incidents informatiques (Wi‑Fi, matériel, logiciels, comptes…).

  • Les étudiants créent des tickets et suivent leurs demandes.
  • Les techniciens traitent les tickets (réponses, changement de statut).
  • Un admin peut superviser (compte/paramètres — minimal).
Objectif pédagogique
  • Projet E5 : authentification (session), BDD, CRUD, droits.
  • Comprendre HTTP : JSON + statuts + tests Postman + Network.
  • Découvrir MVC : code organisé et lisible.
  • Ajouter une “dose” de JS (fetch + DOM) sans faire une SPA.
2) Cas d’utilisation (attendus)
Rôles
ETUDIANT TECH ADMIN
Cas ETUDIANT
  • Se connecter / se déconnecter
  • Créer un ticket (titre, description, priorité, catégorie)
  • Voir la liste de ses tickets
  • Voir le détail d’un ticket
  • Ajouter un message sur son ticket
Cas TECH
  • Se connecter / se déconnecter
  • Voir tous les tickets
  • Filtrer les tickets (statut, priorité)
  • Répondre à un ticket
  • Changer le statut (OPEN → IN_PROGRESS → RESOLVED)
Cas ADMIN (minimal)
  • Se connecter / se déconnecter
  • Accéder à un tableau de bord (placeholder S0)
  • (Option) gérer catégories / comptes (si temps)
Règle de sécurité incontournable
Un étudiant ne doit jamais pouvoir consulter/modifier le ticket d’un autre étudiant (contrôle côté serveur).
3) Architecture choisie (MVC + API)
On garde le même projet, mais organisé
  • Front SSR : pages PHP rendues côté serveur (Bootstrap) via des Controllers.
  • API JSON : endpoints PHP (Controllers API) qui renvoient du JSON.
  • JS : 3–4 fetch ciblés qui consomment l’API et mettent à jour le DOM.
  • Models : accès BDD via PDO (requêtes préparées) + méthodes (find/create/update…).
Ce qu’on ne fait pas
  • Pas de SPA complète
  • Pas de framework front
  • Pas de dizaines d’endpoints
4) Flux de fonctionnement
4.1 Front SSR (pages PHP)
Navigateur → /tickets
   ↓
Controller (TicketController@index)
   ↓
Model (TicketModel::findForUser)
   ↓
View (views/tickets/index.php)
   ↓
HTML Bootstrap renvoyé au navigateur
4.2 API JSON (fetch)
Navigateur (JS fetch) → /api/tickets?status=OPEN
   ↓
API Controller (Api/TicketApiController@index)
   ↓
Model (TicketModel::findForUser + filtre)
   ↓
Réponse JSON + code HTTP (200/401/403/...)
Sessions
L’authentification repose sur une session (cookie). En JS : fetch(..., { credentials: "include" })

Pourquoi cette structure est “E5 solide” ?
  • Lisible : on sait où mettre chaque chose.
  • Évolutif : on ajoute des vues / endpoints sans casser le reste.
  • Testable : Postman sur l’API + navigation sur le front.
6) Responsabilités : Model / View / Controller
Couche Responsabilité Exemples
Model Accès aux données (PDO) + méthodes (find/create/update) TicketModel::findForUser()
View Affichage HTML/Bootstrap avec variables PHP views/tickets/index.php
Controller Reçoit la requête, appelle le model, choisit vue ou JSON TicketController@index, Api/TicketApiController@updateStatus
Règle
Le Controller ne doit pas contenir de SQL : le SQL est dans le Model.
7) “Dose JS” : 3–4 fetch (ciblés)
Fetch #1 — GET /api/me
Afficher nom + rôle dans la navbar
  • Au chargement de la page
  • JS met à jour le DOM
Fetch #2 — GET /api/tickets?status=...
Filtrer la liste sans recharger
  • Event change sur un select
  • Remplace la liste dans le DOM
Fetch #3 — PUT /api/tickets/{id}
Changer statut (TECH) sans recharger
  • Bouton “IN_PROGRESS” / “RESOLVED”
  • Update badge statut dans le DOM
Fetch #4 (bonus) — POST /api/tickets
Créer ticket en AJAX (optionnel)
  • Si la création SSR est déjà OK, ce fetch est un bonus
Minimum JS attendu
  • querySelector, addEventListener, innerHTML
  • fetch + async/await
  • response.json()
  • credentials: "include" (sessions)
8) Étapes de réalisation (S0 → S2/S3)
S0 (4h) — Socle MVC
  • BDD : schema.sql + seed.sql
  • Core : Database + Auth + Router
  • AuthController + View login
  • Dashboards par rôle (views + controller)
  • Sécurité : hash, prepare, htmlspecialchars, session_regenerate_id
S1 (4h) — Tickets SSR + messages + 2 fetch
  • TicketController : index + show + store
  • TicketModel : findForUser + create
  • MessageModel : list + create
  • Fetch #1 (/api/me) + Fetch #2 (filtre)
S2 (4h) — TECH : statut + API PUT
  • Api/TicketApiController : updateStatus (PUT)
  • Contrôle rôle TECH côté serveur
  • Fetch #3 (PUT) + update DOM
  • Preuves Postman + Network
S3 (optionnel) — Bonus + stabilisation
  • Fetch #4 (POST) + rafraîchissement liste
  • Recherche / filtre priorité
  • Finitions + entraînement oral
9) Points cybersécurité à intégrer
À appliquer dans le projet
  • Mots de passe : hash (password_hash/verify), jamais en clair
  • SQL : requêtes préparées PDO, pas de concaténation
  • XSS : htmlspecialchars sur toutes les sorties
  • Session : regenerate_id après login + logout propre
  • Accès : requireLogin + requireRole sur pages et API
CSRF (bonus)
Option : token CSRF dans les formulaires si vous avez le temps.
